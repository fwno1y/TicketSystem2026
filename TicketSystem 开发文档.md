#  TicketSystem 开发文档

这是fwno1y的TicketSystem

---

## 1. 系统的模块划分


### 1.1 命令解析与交互层
系统以标准输入输出（I/O）作为媒介。`main.cpp` 通过 `std::getline` 持续输入命令行，并移交 `CommandParser` 处理：
*   **指令提取**：解析带有 `[timestamp]` 标签的命令，将其拆分为对应的指令字符串
*   **参数处理**： `Command` 结构体维护了一个定长的布尔数组及字符串数组，将 `-c`、`-u` 等代表修饰符的 key 快速映射为对应的参数。

### 1.2 业务逻辑管理层
主要由三个管理器实现：
*   **`UserManager`**：负责处理用户相关指令。它管理用户权限比较逻辑，维护用户登录状态表（在内存中缓存），控制用户的 profile 获取和修改逻辑。
*   **`TrainManager`**：负责解析复杂的车次参数，换算从终点到各站的前缀价格及区间旅行时间，调配区间的余票扣减，以及执行票务排序查询和两车换乘的规划。
*   **`OrderManager`**：不仅负责将购买记录持久化，以及订单的状态流转控制，包括对候补订单的登记，退票时对排队队列的处理

### 1.3 数据结构层
*   **`sjtu::vector`**：见STLite-ACM-2026
*   **`sjtu::map`**：见STLite-ACM-2026

### 1.4 物理存储与磁盘索引层

*   **`MemoryRiver`**：大块定长数据的管理器，用于记录用户、车次、订单和余票的数据。
*   **`BPlusTree`**：用以维护映射（如用户名、车次、站点到文件偏移量 `pos` 的映射）。每个 B+ 树节点都是定长的并驻留在磁盘中，避免将索引树整体读入内存。

## 2. 底层存储设计


### 2.1 存储：`MemoryRiver` 
`MemoryRiver` 类是为了方便对文件的操作而封装了一些操作。该类的功能有

* 与一个文件名（string）绑定
* 往文件中写入一个int或T类型对象
* 往文件中读取一个int或T类型对象（T的大小大于int）

### 2.2 索引：磁盘 B+ 树

本项目中建立的 B+ 树实例如下表：

| B+ 树文件名称 | 键 (Key) 的构成结构与比较规则 | 值 (Value) 的含义 | 作用 |
| :--- | :--- | :--- | :--- |
| `user_index` | `UserKey` (包含 `char username[21]`) | `int` (指向用户实体在文件中的位置) | 查询/更改用户资料时快速寻址定位 |
| `train_index` | `TrainKey` (包含 `char trainID[21]`) | `int` (指向车次实体在文件中的位置) | 判断车次是否存在、快速拉取车次主干属性 |
| `train_date_index` | `DayKey` (包含 `trainID` + 始发 `Date`) | `int` (指向该天该车次余票在文件中的位置) | 定位并读取具体某一天发车的余票 `TicketLeft` |
| `train_station_index` | `StationKey` (包含中途站 `stations` 和 `trainID`) | `Station` (列车中的车站索引，包含物理下标) | 车票查询和中转中，快速反查有哪些列车经停某站并在第几站 |
| `order_index` | `OrderKey` (包含 `username` + 自增 `orderID`) | `int` (指向订单实体在文件中的位置) | 列表输出所有订单，因包含 orderID 升序，所以能快速完成从新到旧的排序 |
| `pending_index` | `PendingOrderKey` (包含 `trainID` + `Date` + `orderID`) | `PendingOrder` (候补订单项信息，包含购票起终点物理下标与张数) | 退票导致余票更新时，作为候补排队队列依时间先后顺序尝试促单 |

## 3. 核心业务处理逻辑与函数实现

### 3.1 用户管理模块 (`UserManager`)

#### `add_user`
*   **逻辑思路**：首先尝试在用户 B+ 树索引中查找该用户名。如果用户 B+ 树为空，说明当前是系统注册的第一个用户，直接将其作为超级管理员，权限级别设置为最高等级（十级），写入用户数据文件并插入 B+ 树索引，返回成功。如果不是第一个用户，则需要验证发起人的登录状态，并从在线列表中获取发起人的安全等级。如果发起人的安全等级严格大于新注册用户的安全等级，并且该用户名在索引中不存在，则将新用户写入数据库，同时记录在用户 B+ 树索引与内存临时缓存中。

#### `login`与`logout`
*   **逻辑思路**：在 `login` 时，首先检查内存的在线用户表，防止重复登录。接着在用户 B+ 树索引中查找用户名以获取其物理存储位置，从数据文件中读取该用户结构体。比对传入密码与数据库中的密码，如果相符，则在内存的在线用户表中记录该用户已登录，并关联其权限等级。在 `logout` 时，直接判定是否在线，若在线则在内存的在线表里擦除该用户名，完成退出。

#### `query_profile`与`modify_profile`
*   **逻辑思路**：在查询用户时，首先检查发起人是否登录并具有比被查询人更高的权限等级（或为同一人）。验证通过后，优先尝试在用户内存缓存中读取目标用户，以减少物理读盘。若缓存未命中，则过索引读盘并载入缓存。在修改用户资料时，除了进行权限溢出校验外，允许选择性更改密码、姓名、邮箱和等级。更新字段后执行文件物理覆写，并同步更新在线用户等级和内存缓存以保证数据一致性。

### 3.2 车次与余票控制中心 (`TrainManager`)

#### `add_train`, `delete_train`与`release_train`
*   **逻辑思路**：`add_train` 检查车次 ID 是否重复，如果不存在，则将车站列表、票价、旅行时间、停站时间等信息转换并写入车次实体文件中，将对应的键值关系记录到车次 B+ 树索引。`delete_train` 用于在未发布状态下安全移除车次关系。`release_train` 将车次的发布状态改为已发布，随后开始为该列车在其整个可运行日期区间的每一天，都在物理票仓中写入一份初始化的余票记录（记录每两个连续站之间的可用座位）；同时，在车站 B+ 树索引中注册该车经过的每一个车站。

#### `try_buy` (预扣票核心函数)
*   **逻辑思路**：此函数为购票提供可靠的票池检查与预扣席位服务。接收到乘车段以及用户的上车日期后，首先根据该中途站出发的绝对分钟偏移，推导出该班次列车必须在哪一天从**始发站**发出。随后通过车次 ID 与始发日期去日期索引树中检索，获取对应的余票记录。检查起终点范围内所有基本站点的余票极小值，如果发现极小值大于等于购买张数，则对该区间内的所有子段全部减去购票张数，写回磁盘，并返回总票价；如果余票不足，直接返回零。

#### `refund_seats`
*   **逻辑思路**：当订单被退票时，此函数用于返还座位。根据订单提供的车次和始发日期定位到具体的余票段，并把退订的张数累加回订购区间所对应的每一个车站子路段中，执行物理覆盖更新，释放座位资源。

---

### 3.3 订单流转与候补排队机制 (`OrderManager`)

#### `buy_ticket`
*   **逻辑思路**：首先检查购买的合法性，若买票用户未登录或车次未发布则直接失败。否则调用 `train_manager.try_buy` 进行座位预扣减。如果购票成功（返回了正价格），则生成一个成功状态的订单并写入文件，建立订单索引。如果由于余票不足失败且用户在参数中指明愿意候补排队，则生成一个候补状态的订单写入文件，同时在候补索引 B+ 树中插入一条排队项（主键由车次、始发日期和自增订单 ID 组成，以确保排队顺序）；若不愿意排队则直接返回购买失败。

#### `query_order`
*   **逻辑思路**：利用订单索引 B+ 树获取属于该用户的所有订单位置。由于主键带有自增的订单 ID，系统只需以逆序形式遍历这些位置偏移并反查订单实体执行打印，即可天然实现“按交易时间由新到旧”的排序展现。

#### `refund_ticket`
*   **逻辑思路**：系统通过查询订单索引，获取用户由新到旧的第 $n$ 个订单。如果该订单已退票，则拒绝；如果是候补订单，则将其在候补索引 B+ 树中的排队券剔除，状态改为已退票写回；如果是成功售出的票，则修改订单状态更新存盘，接着调用 `train_manager.refund_seats` 释放原占用的座位，并在最后调用排队处理函数 `process_pending`。

#### `process_pending` (候补实现)
*   **逻辑思路**：当有订单完成退票，座位资源得到释放后，此函数会被触发。由于候补 B+ 树索引的 Key 以订单 ID 结尾，系统调出的该发车日期的所有排队等候群，天然是按**下单时间先后顺序**排列的。系统从最早排队的订单开始滚动审查：根据其车程段读取最新的区间余票，如果发现当前余票能够足额满足其购票数量，便在票仓扣除该数量，更新其订单状态为成功，并从排队树上移除该项；若余票不能满足，系统不会发生阻塞，而是**继续向下查看后续排队项目，如果后续排队所需的票数较少且余票正好能够满足，则优先促成该后续项目**。循环结束后整合写入磁盘中。

## 4. 项目类设计

为保证可以直接写入文件，自定义的所有结构体与类均控制为定长：

### 4.1 用户类 `User`
```cpp
class User {
public:
    char username[21]{};     // 用户唯一账户名，末尾备足'\0'防止截断溢出
    char password[31]{};     // 可见密码
    char name[16]{};         // 真实姓名（2至5个汉字，UTF-8三字节编码）
    char mailAddr[31]{};     // 邮箱
    int privilege = 0;       // 权限优先级级别

    User() = default;
    User(const std::string& u, const std::string& p, const std::string& n, const std::string& m, int g);
    ~User();
};
```

### 4.2 车次与区间余票类 `Train` 与 `TicketLeft`
```cpp
class Train {
public:
    char trainID[21]{};               // 车次主ID键
    int stationNum = 0;               // 经停站数
    int seatNum = 0;                  // 单车初始最大座位座位容量
    char stations[MAX_STATION][31]{}; // 本车所属的所有站名列表（最多经停100个车站）
    int prices[MAX_STATION]{};        // 区间价格前缀和，计算AB两站间价格只需在prices做差
    Time startTime;                   // 始发相对出发时刻
    int arriveTimes[MAX_STATION]{};   // 累计旅行分钟（始发的相对事件累计）
    int leaveTimes[MAX_STATION]{};    // 累计出发分钟（始发的相对事件累计）
    Date saleDateStart;               // 售发起始
    Date saleDateEnd;                 // 售发截止
    char type{};                      // 车次大写字母类型
    bool released = false;            // 是否已发布
    int ticket_pos = -1;              // 票轨对应的文件第一天索引偏移，用以逻辑加速
};

struct TicketLeft {
    int tickets[MAX_STATION];         // 存段票据。tickets[i] 代表在第 i 站至第 i+1 站间剩余在座席数。
    TicketLeft() = default;
    TicketLeft(int seat, int station);
};
```

### 4.3 订单与候补类 `Order` 与 `PendingOrder`
```cpp
class Order {
public:
    int orderID = 0;                  // 全局自增独立订单ID号
    int timestamp = 0;                // 下单时间戳记
    OrderStatus status;               // 订单状态：SUCCESS, PENDING, REFUNDED
    char username[21]{};
    char trainID[21]{};
    Date start_date;                  // 计算出的物理始发日期，用于退票反查
    Date user_date;                   // 上车站车票上的日期
    char from_station[31]{};
    char to_station[31]{};
    int from_idx = 0;                 // 出发站的在车轨上的物理下标序号
    int to_idx = 0;                   // 到达站在车次上的物理下标序号
    DateTime leave_time;              // 出发时刻
    DateTime arrive_time;             // 到达时刻
    int price = 0;                    // 总单价
    int num = 0;                      // 购买车单张数
};

struct PendingOrder {
    int order_pos = -1;               // 订单在 order_data 物理堆文件里的实际存址偏移量
    int orderID = 0;                  // 排队的等候券ID，用来做顺序标志
    int from_idx = 0;                 // 出发站物理下标
    int to_idx = 0;                   // 到达站物理下标
    int num = 0;                      // 排队购票数量
};
```
